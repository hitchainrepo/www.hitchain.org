HitChain Project Biweekly Report:17 December—29 December, 2018

December 29, 2018

1. Project Progress

1) Configure another foundation charity server node, and form into a dual service node network.

2) Separate the server IPFS and client IPFS, and save the code into different repositories for separate follow-up updates：

a、Remove the timed report function for server IPFS；

b、Remove the permission query operation when executing “ipfs add” operation, because the server itself is an authority controller.

3) Separate the dependent gx repository of server and client IPFS for follow-up updates. (gx repository for client: https://github.com/nigel007/gx; gx repository for server: https://github.com/nigel007/gx_server)

a、Remove the limitation of adding parameters when initializing ipfs for server and client. For client, it can get the server list from the background server；

b、Remove the limitation of adding event listener to disaster recovery server, for the server itself is a listener.

4) Update something for client IPFS:

a、Search all the Hit servers when initializing；

b、After getting the Hit server list, the client IPFS will get the speed to all the servers through ping command. Meanwhile, the locations of servers will be shown on the commander；

c、 When initializing, the client can choose its own Hit network access point according to servers’ locations and speed.

5) Find the default configuration for storage in IPFS, and we can modify the storage capacity manually.

6) Add Git file index to Hitbucket, and add sha1 and ipfs-hash to the git index file to resolve the duplicate file update issue.

7) Add Ethereum account support to Hitbucket to receive the HIT.

8) Add hash record feature to Hitbucket to record the repository hash value. This feature will change to publish the hash to the ethereum.
